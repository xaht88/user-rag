# Техническое задание: RAG Веб-приложение

## 1. Описание проекта

RAG (Retrieval-Augmented Generation) веб-приложение для работы с документами через диалоговый интерфейс. Система загружает документы, создаёт векторные эмбеддинги, выполняет семантический поиск по запросу пользователя и генерирует ответ через выбранную LLM с указанием источников.

---

## 2. Целевая аудитория

- Аналитики, юристы, исследователи — работа с объёмными PDF-документами
- Разработчики — изучение технической документации
- Мобильные пользователи — доступ с телефона

---

## 3. Функциональные требования

### 3.1 Загрузка и обработка документов

| ID | Требование |
|----|-----------|
| F-01 | Поддержка форматов: PDF, DOCX, TXT, MD |
| F-02 | Максимальный размер файла: 50 МБ |
| F-03 | Разбивка на чанки: 512 токенов, перекрытие 50 токенов |
| F-04 | Создание эмбеддингов через BERT-based модель (sentence-transformers) |
| F-05 | Сохранение в Vector Store (ChromaDB) |
| F-06 | Отображение прогресса обработки и итогового статуса |
| F-07 | Хранение метаданных: имя файла, дата, кол-во страниц, кол-во чанков |

### 3.2 Выбор LLM

| ID | Требование |
|----|-----------|
| F-08 | Список доступных LLM из конфигурации |
| F-09 | Провайдеры: OpenAI (GPT-4o, GPT-4o-mini), Ollama (локальные модели) |
| F-10 | Сохранение выбора в рамках сессии |
| F-11 | Поддержка пользовательского API-ключа (только в памяти сессии) |
| F-12 | Уведомление при недоступности выбранной модели |

### 3.3 Диалоговый интерфейс

| ID | Требование |
|----|-----------|
| F-13 | Векторный поиск топ-5 релевантных чанков |
| F-14 | Формирование Context Window и передача в LLM |
| F-15 | Отображение ответа в течение 30 секунд |
| F-16 | Отображение источников: имя файла + номер страницы |
| F-17 | История диалога в рамках сессии |
| F-18 | Индикатор загрузки при обработке запроса |

### 3.4 Управление документами

| ID | Требование |
|----|-----------|
| F-19 | Список загруженных документов с метаданными |
| F-20 | Удаление документа с очисткой чанков из Vector Store (≤ 5 сек) |
| F-21 | Выбор активных документов для поиска |
| F-22 | Лимит: не более 10 документов на сессию |

## 4. Нефункциональные требования

| ID | Требование |
|----|-----------|
| NF-01 | Время ответа LLM: ≤ 30 секунд |
| NF-02 | Время обработки документа: ≤ 60 секунд |

## 5. Обработка ошибок

### 5.1 Загрузка файлов

| Сценарий | UX-решение |
|----------|-----------|
| Файл > 50 МБ | Toast-уведомление: «Файл превышает 50 МБ» |
| Неподдерживаемый формат | Toast: «Допустимые форматы: PDF, DOCX, TXT, MD» |
| Повреждённый файл | Toast с описанием ошибки парсинга |
| Сетевая ошибка при загрузке | Toast: «Ошибка сети. Попробуйте снова» + кнопка повтора |

### 5.2 LLM и запросы

| Сценарий | UX-решение |
|----------|-----------|
| LLM недоступна | Inline-ошибка в чате + список доступных альтернатив |
| Тайм-аут (> 30 сек) | Сообщение в чате: «Превышено время ожидания» + кнопка повтора |
| Нет загруженных документов | Подсказка в поле ввода: «Загрузите документ для начала» |
| Неверный API-ключ | Inline-ошибка: «Неверный API-ключ. Проверьте настройки» |

### 5.3 Управление документами

| Сценарий | UX-решение |
|----------|-----------|
| Ошибка удаления | Toast: «Не удалось удалить документ» + повтор |
| Превышен лимит (10 документов) | Toast: «Достигнут лимит. Удалите документ для загрузки нового» |

## 6. Архитектура (рекомендуемая)

```
┌─────────────────────────────────────────────────────┐
│                    Frontend (React/Next.js)          │
│  AppLayout │ DocumentPanel │ ChatPanel │ LLMSelector │
└──────────────────────┬──────────────────────────────┘
                       │ REST API / WebSocket
┌──────────────────────▼──────────────────────────────┐
│                  Backend (FastAPI / Python)          │
│                                                     │
│  /upload  →  Document_Processor  →  Embedding_Model │
│  /query   →  Vector_Store (ChromaDB)                │
│           →  LLM_Connector (OpenAI / Ollama)        │
│  /session →  Session Manager                        │
└─────────────────────────────────────────────────────┘
```

### Компоненты Backend

- `document_processor.py` — парсинг PDF/DOCX/TXT/MD, чанкинг
- `embedding_service.py` — создание эмбеддингов (sentence-transformers)
- `vector_store.py` — CRUD операции с ChromaDB
- `llm_connector.py` — абстракция над OpenAI и Ollama
- `session_manager.py` — управление сессиями и изоляция данных
- `api/routes.py` — FastAPI роуты

### Компоненты Frontend

- `AppLayout` — двухпанельный responsive layout
- `DocumentPanel` — загрузка и список документов
- `ChatPanel` — история диалога, поле ввода
- `LLMSelector` — dropdown выбора модели
- `SourceCard` — карточка источника

---

## 7. Критерии готовности (Definition of Done)

- [ ] Загрузка PDF/DOCX/TXT/MD работает без ошибок
- [ ] Эмбеддинги создаются и сохраняются в ChromaDB
- [ ] Векторный поиск возвращает топ-5 релевантных чанков
- [ ] Ответ LLM отображается с источниками
- [ ] Интерфейс адаптирован для мобильных устройств
- [ ] Все ошибки обрабатываются с понятными сообщениями
- [ ] Данные сессии изолированы и очищаются при закрытии
- [ ] Время ответа ≤ 30 секунд на тестовых документах

## 8. Решения по реализации Frontend (утверждено)

- **Frontend stack**: Next.js (App Router) + TypeScript + Tailwind CSS
- **UI-концепция**: Split-Panel (`DocumentPanel` + `ChatPanel`) с мобильным fallback
- **Интеграция с backend**:
  - dev: frontend поднимается отдельно (`next dev`), backend (`uvicorn`)
  - build: статика публикуется в `rag_app/backend/static/next`, template в `rag_app/backend/templates`
- **Состояния интерфейса**: для каждого ключевого сценария должны быть реализованы `loading`, `error`, `success`

### 8.1 Минимум 3 обязательные функции Frontend

| ID | Функция | Критерий приемки |
|----|---------|------------------|
| FE-01 | Upload документов | Пользователь может загрузить `pdf/docx/txt/md`, увидеть прогресс и итоговый статус |
| FE-02 | Chat с источниками | Ответ в чате отображается с источниками и состояниями `loading/error` |
| FE-03 | Выбор LLM | Выбранный провайдер/модель применяется к запросу и отображается в интерфейсе |

### 8.2 Структура frontend-слоев

- `app/` — страницы и layout
- `components/` — UI-компоненты (`AppLayout`, `DocumentPanel`, `ChatPanel`, `LLMSelector`, `SourceCard`)
- `features/` — прикладная логика (`documents`, `chat`, `llm`)
- `shared/` — api client, типы, утилиты

### 8.3 Автотесты (минимальные)

- Тест 1: `DocumentPanel` корректно показывает `loading/error/success`
- Тест 2: `ChatPanel` отправляет сообщение и отображает ошибку при невалидном ответе
- Тест 3: `LLMSelector` обновляет конфигурацию модели и передает ее в callback
